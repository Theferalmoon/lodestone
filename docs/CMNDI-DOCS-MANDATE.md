# CMNDI Documentation Mandate (CMNDI-DOCS-MANDATE-001)

> **Status:** Permanent CMNDI mandate. Effective 2026-05-02.
> **Owner:** All CMNDI repo maintainers.
> **Enforcement:** Post-commit hook + warn-tier pre-commit gate + CI assertion.
> **Revision:** 2026-05-02 — relaxed per devil's-advocate review (tier-based scope, warn-by-default gate, IP-summary public-repo carve-out).

---

## 1. Why this mandate exists

CMNDI tooling has accumulated 75+ services, 9 compliance frameworks, and
multiple SaaS products. Without a fixed doc surface per repo, operators,
friends, exec audiences, and AI agents (cmndclaw, claude, etc.) all read
different sources of truth and drift.

This mandate establishes a **tiered, multi-audience documentation set** that
lives in every CMNDI repo, is kept current by trigger, and is shipped as a
single self-contained HTML for handoff/exec/customer use.

## 2. Required documents — by repo tier

CMNDI repos fall into two tiers. Each tier has a different required doc set:

### Tier C — Customer-facing repos (full 6 required)

Repos that ship to friends, alpha customers, or have a public surface.
ALL six source markdown files plus the generated HTML are required.

| File | Audience | Length target | Purpose |
|---|---|---|---|
| `01-executive-1pager.md` | C-suite / investors | 1 page | What it is, who it's for, why it matters, dollar implications |
| `02-technical-spec.md` | Engineers / agents | unbounded | Full architecture, APIs, data flows, dependencies, applicable mandates |
| `03-end-user-guide.md` | Operators / friends | tutorial-length | Install, configure, daily use, troubleshooting |
| `04-brochure.md` | Sales / marketing | 2-3 pages | Feature highlights, comparison, use cases, calls to action |
| `05-talking-points.md` | Demos / interviews | bullet list | Quick sound bites for pitches, Q&A prep |
| `06-ip-summary.md` | Legal / IP / strategy | 1-2 pages | Inventive claims, prior art delta, license posture, trademark, defensible moats |

**Tier C list:** `lodestone` (post-v0.1.2), `windrose-helm` (at first ship),
`ulysses-s-grants-saas`, `HAL-Hygiene-Assessment-Logic`, future Forge.

### Tier I — Internal infrastructure repos (2 required)

Repos that don't ship externally. Only the two engineering-facing docs are
required; the marketing/legal four are OPTIONAL but recommended for repos
that touch revenue or recruiting paths.

| File | Audience | Required? |
|---|---|---|
| `02-technical-spec.md` | Engineers / agents | **Required** |
| `03-end-user-guide.md` | Operators | **Required** |
| `01-executive-1pager.md` | C-suite | Optional |
| `04-brochure.md`, `05-talking-points.md`, `06-ip-summary.md` | Sales / legal | Optional |

**Tier I list:** `local-opus-lab` (HQ), `CMNDClaw`, `cmmc-scanner` (PRIVATE
forever per 2026-04-27 mandate), tooling repos.

### Generated artifact (both tiers)

- `index.html` — **single self-contained file** with all PRESENT docs,
  interactive TOC, anchor links, search filter, CMNDI dark-SOC styling.
  No external dependencies (no CDN; CSS/JS inline). Generated by
  `scripts/docs/cmndi-docs-build.py`. Never hand-edited.

### Public-repo IP-summary carve-out

Repos with a public-facing license (Apache 2.0, MIT, etc.) MUST NOT include
`06-ip-summary.md`. Inventive claims placed in a public repo become public
record and can create unintended commitments + give prior-art targets to
opposers. Keep IP-summary in private repos only.

Affected: `lodestone` (Apache 2.0) — REQUIRED docs are 1-5 only; IP summary
lives in `local-opus-lab/docs/cmndi/06-ip-summary.md` instead.

## 3. Update trigger (mandatory automation)

A post-commit hook MUST run on every commit and:

1. Detect whether the commit touched any tracked file outside `docs/cmndi/index.html`
2. If yes: regenerate `index.html` from the present source files
3. If the regenerated `index.html` differs from the previous, leave the
   working tree dirty so the operator stages + commits the refresh
4. Stale-docs warning fires when:
   - A user-visible feature release tag (semver-bump like `v[0-9].[0-9].[0-9]`)
     is created AND `docs/cmndi/0*.md` was not touched in the commits
     since the previous release tag.
   - This replaces the previous "20 commits" rule, which was arbitrary and
     fired on refactor sprints that don't need docs refreshes.

Reference implementation: `scripts/docs/cmndi-docs-hook.sh`
Build script: `scripts/docs/cmndi-docs-build.py`

## 4. Tooling enhancement requirement (scoped to NEW claims)

Authors MUST use the full CMNDI tooling stack to verify any **new**
technical, architectural, compliance, or IP claim added to a doc. This
requirement is SCOPED — it does NOT fire on typo fixes, formatting changes,
broken-link repairs, or cosmetic edits.

When the requirement DOES fire (new claim, new section, new component):

- **MCP / KG queries** — every claim about service topology, mandates,
  inventive history, or compliance posture MUST be cross-checked against
  the live KG via `bash scripts/check-mandates.sh "<topic>"` and/or
  `mcp__plugin_claude-mem_mcp-search__smart_search`. Cite the anchor.
- **Context-engine retrieval** — for any architectural claim, query the
  context-engine to confirm the current code matches the doc.
- **CMDB lookup** — for any service mention, verify the CI exists and the
  `architecture_spec` matches the doc claim.
- **Plugin chain** — invoke domain-relevant Claude Code plugins where
  applicable: `claude-code-guide` for SDK/Agent claims, `microsoft-docs`
  for Azure/MS surface, `cloudflare-docs` for CF surface, `huggingface-skills`
  for model claims, `executive-mentor` for the IP-summary devil's-advocate
  pass, `code-reviewer` for any embedded code samples.
- **Subagent dispatch** — for technical-spec sections that span more than
  one service, dispatch an `Explore` or `general-purpose` subagent in
  parallel to verify the cross-cuts before writing.
- **External research** — Tavily + Brave Search are MANDATORY for all
  external-fact claims (per the permanent CMNDI mandate in CLAUDE.md §5).

## 4b. Compliance scan gate — warn-by-default + hard-fail on banned vendors

Pre-commit gate runs the CMNDI compliance tool chain. Default behavior is
**WARN** for most findings; only one class is HARD-FAIL:

### Hard-fail (blocks commit, no exemption)

- **Banned-vendor introduction** in the staged diff (Qwen, DeepSeek,
  Baichuan, ChatGLM, InternLM, Yi, Yandex, Sber, MiniMax, Skywork,
  Moonshot, Hunyuan). The PreToolUse Bash hook only catches direct
  invocations — pre-commit catches what slips in via Edit/Write.

### Warn-tier (visible, not blocking)

- **rick-hays-bot findings** on staged files — surfaced; operator decides
- **CIS / NIST / CMMC / ISO scanner findings** — surfaced
- **`scripts/compliance/verify-airgap.sh`** results — surfaced for
  network-egress changes
- **`scripts/check-mandates.sh "<topic>"`** retrievals — surfaced for the
  touched topic

### Documented-finding override

Inline `// HAYS-WAIVED: <reason>` (or `# HAYS-WAIVED:` for non-C-family
syntaxes) suppresses the warning and emits an audit-log line. The reason
field is required and SHOULD reference an issue or runbook. Waivers
accumulate to a per-repo registry at `docs/cmndi/hays-waivers.md` reviewed
quarterly.

### Hotfix exemption

Branches matching `hotfix/*`, `incident/*`, or `revert/*` skip the
warn-tier surfacing entirely; only banned-vendor hard-fail still applies.
Rationale: incident response can't be gated on noisy compliance output.

### Provenance + dep-file gate

Dependency files (package.json, Cargo.toml, etc.) require `clear-prov`
token before commit — this is unchanged from existing behavior, NOT new.

Reference implementation: `scripts/docs/cmndi-compliance-gate.sh`

## 5. CI integration

GitHub-hosted CMNDI repos SHOULD add a workflow step that:

- Re-runs the build
- `git diff --exit-code docs/cmndi/index.html` — fails the build if HTML is
  stale relative to source MD
- Forces operator to run the build locally and commit the refresh before merge

## Appendix: how to add this to a new repo

```bash
# From the new repo root:
mkdir -p docs/cmndi scripts/docs

# Tier I (internal): copy 02 + 03 only
cp /home/theferalmoon/local-opus-lab/docs/cmndi/02-technical-spec.md docs/cmndi/
cp /home/theferalmoon/local-opus-lab/docs/cmndi/03-end-user-guide.md docs/cmndi/

# Tier C (customer-facing): copy 01-05 (and 06 only if PRIVATE)
cp /home/theferalmoon/local-opus-lab/docs/cmndi/0[1-5]-*.md docs/cmndi/

# Build + hook scripts (always)
cp /home/theferalmoon/local-opus-lab/scripts/docs/cmndi-docs-build.py scripts/docs/
cp /home/theferalmoon/local-opus-lab/scripts/docs/cmndi-docs-hook.sh scripts/docs/
cp /home/theferalmoon/local-opus-lab/scripts/docs/cmndi-compliance-gate.sh scripts/docs/

# Wire hooks: append the cmndi-docs-hook line to .git/hooks/post-commit
# and the cmndi-compliance-gate line to .git/hooks/pre-commit
# (see local-opus-lab/.git/hooks/ for reference)

# Edit the doc stubs to reflect THIS repo's specifics
# Run the build to verify: python3 scripts/docs/cmndi-docs-build.py
# Update §9 retro-scan registry in docs/CMNDI-DOCS-MANDATE.md to mark "active"
```
